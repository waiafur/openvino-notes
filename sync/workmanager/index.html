    Модуль синхронизации: Фоновые задачи (WorkManager)

    Для обеспечения гарантированной доставки изменений на сервер даже в случае закрытия приложения пользователем, вся инфраструктура запуска синхронизации вынесена из главного процесса UI в системный планировщик Jetpack WorkManager.

    1. Планирование задач через WorkManagerSyncScheduler

    Запуск фонового процесса инкапсулирован в классе WorkManagerSyncScheduler, реализующем доменный интерфейс SyncScheduler. Вместо хаотичного вызова корутин из слоев интерфейса (ViewModel), отправка задачи триггерится системными намерениями.

    Критерии и ограничения запуска (Constraints)

    Для минимизации расхода заряда батареи и исключения ошибок холостых сетевых запросов настраиваются жесткие системные ограничения:

    val constraints = Constraints.Builder()
    .setRequiredNetworkType(NetworkType.CONNECTED) // Обязательное наличие интернета
    .build()

    Обеспечение уникальности выполнения

    Синхронизация тяжелых файлов и JSON-структур требовательна к транзакционности. Для предотвращения состояния гонки (Race Condition), когда несколько триггеров пытаются одновременно писать данные в одну и ту же учетную запись Firebase, используется механизм уникальных воркеров: 

    workManager.enqueueUniqueWork(
    "sync_work_$userId",            // Уникальное имя воркера, привязанное к UID пользователя
    ExistingWorkPolicy.REPLACE,     // Стратегия: если прошлый воркер еще работает, заменить его новым
    syncRequest
)
    Использование стратегии ExistingWorkPolicy.REPLACE гарантирует, что если пользователь совершил серию быстрых изменений в заметках, цепочка промежуточных воркеров прервется и запустится один актуальный процесс, содержащий последний слепок локальной БД.

    2. Реализация SyncWorker и обработка жизненного цикла

    Класс SyncWorker является точкой входа, активируемой операционной системой Android при выполнении заданных Constraints. Он наследуется от CoroutineWorker, предоставляя встроенную поддержку асинхронных операций Kotlin Coroutines.Порядок извлечения контекста:Воркер пытается извлечь USER_ID из переданного inputData.Если в inputData идентификатор отсутствует (например, при автоматическом перезапуске системой), воркер обращается к AuthManager.getCurrentUserId().Если пользователь выполнил деавторизацию (Sign-Out), выполнение мгновенно прерывается с результатом Result.failure(), предотвращая утечку или смешивание данных.

    3. Политика обработки ошибок и перезапусков (Retry Policy)

    Внутри метода doWork() весь блок синхронизации обернут в защитный контейнер обработки исключений. В зависимости от природы возникшей ошибки SyncWorker дифференцирует возвращаемый операционной системе результат:

    Тип исключения / Сценарий Возвращаемый результат Поведение системы
    Успешное выполнение Result.success() Задача завершается. Локальные данные полностью консистентны с облаком Firebase.
    CancellationException throw e Сигнализирует о том, что корутина была отменена изнутри приложения (например, принудительное закрытие процесса). Перевыбрасывается для корректной остановки воркера.
    IOException (Сбой сети, тайм-аут сокета, падение сервера Firebase) Result.retry() Повтор выполнения. WorkManager ставит задачу в очередь на повторный запуск. Применяется экспоненциальная политика задержки (Exponential Backoff), увеличивающая интервал между попытками, чтобы не перегружать сеть устройства.
    SerializationException / NullPointerException (Ошибка парсинга JSON, битая структура данных БД) Result.failure() Критический сбой. Задача прерывается без права на автоматический повтор. Это защищает систему от бесконечного цикла перезапусков воркера, который привел бы к быстрой разрядке батареи из-за заведомо невыполнимой задачи. 
    try {
    syncManager.sync(userId)
    Result.success()
} catch (e: CancellationException) {
    throw e
} catch (e: java.io.IOException) {
    Timber.e(e, "Sync retryable error: %s", e.message)
    Result.retry() // Сетевая ошибка, повторить позже
} catch (e: Exception) {
    Timber.e(e, "Sync fatal error: %s", e.message)
    Result.failure() // Критическая ошибка логики, остановить выполнение
}

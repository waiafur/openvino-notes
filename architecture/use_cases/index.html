    Бизнес-логика и Сценарии использования (Use Cases)

    В соответствии с принципами Clean Architecture, слой бизнес-логики (Domain) полностью изолирован от деталей реализации UI (Jetpack Compose) и источников данных (Room/Firebase). Вся логика приложения инкапсулирована в точечных классах сценариев — Use Cases (Интеракторах).

    Такой подход гарантирует протестированность кода (через MockK) и позволяет легко изменять поведение приложения без изменения UI-компонентов.

    1. Структура Use Cases в DI-контейнере

    Все сценарии использования регистрируются в AppModule.kt через Koin и группируются в общую структуру-фасад NotesUseCases для удобной инжекции в NotesViewModel.

    Сценарии для управления папками (Folder Use Cases)

    Класс Use Case Область ответственности Особенности реализации
    CreateFolderUseCase Создание новой папки Генерирует UUID, проставляет дефолтные таймстампы.
    ObserveFoldersUseCase Реактивный стрим папок Возвращает Flow<List<NoteFolder>> для UI в реальном времени.
    GetFolderUseCase Получение папки по ID Запрашивает конкретную папку из репозитория.
    UpdateFolderUseCase Обновление папок Используется при изменении метаданных папки.
    DeleteFolderUseCase Удаление папки Реализует логику мягкого удаления перед синхронизацией.

    Сценарии для управления заметками (Note Use Cases)

    Центральный набор сценариев, покрывающий весь жизненный цикл текстового контента.

    Класс Use Case Область ответственности Особенности реализации
    CreateNoteUseCase Инициализация новой заметки Создает пустой или предзаполненный объект Note.
    GetNoteUseCase Чтение заметки по ID Вызывается при переходе на экран редактора (EditorViewModel).
    UpdateNoteUseCase Сохранение изменений контента Обновляет updatedAt и сбрасывает флаг isSynced.
    DeleteNoteUseCase Удаление заметки Маркирует isDeleted = true для отложенной очистки.
    ObserveNotesUseCase Стрим всех заметок Возвращает реактивный список для общего экрана.
    ObserveNotesByFolderUseCase Фильтрация заметок по папке Выбирает только те элементы, где folderId == targetId.
    MoveNoteToFolderUseCase Перемещение заметки Изменяет привязку folderId у существующей заметки.
    SearchNotesUseCase Локальный поиск по контенту Фильтрует Flow заметок по совпадению в заголовке/тексте.
    SwitchFavoriteUseCase Переключение инверсии "Избранного" Атомарно меняет флаг isFavorite (0/1).
    GetAllFavoritesUseCase Выборка избранного контента Формирует список для быстрой навигации на главном экране.
    ValidateDuplicateNoteTitleUseCase Валидация заголовков Исключает создание заметок с одинаковыми именами в одной папке.
    GetUserIdUseCase Получение UID текущего юзера Запрашивает Firebase Auth через AuthRepository.

    2. Специальные сервисные сценарии

    Жизненный цикл сессии: ClearLocalDataOnSignOut

    Особый интерактор безопасности, который предотвращает утечку персональных данных при смене аккаунта.

    • Триггер: Вызывается внутри AuthViewModel при ручном выходе пользователя из системы (Sign Out).
    • Поведение: Атомарно очищает локальные таблицы Room (notes, folders, media) до сброса текущей сессии в Firebase Auth, чтобы новый пользователь не увидел кэш предыдущего.
    • Зависимости: Агрегирует в себе ObserveNotesUseCase, DeleteNoteUseCase, ObserveFoldersUseCase и DeleteFolderUseCase для каскадной зачистки.

    3. Схема взаимодействия слоев (Data Flow)

    Любое действие пользователя в UI проходит строго регламентированный путь:

    1. UI (Compose Screen) перехватывает нажатие (например, клик на "Добавить в избранное").
    2. ViewModel вызывает соответствующий Use Case: useCases.switchFavoriteUseCase(noteId).
    3. Use Case содержит бизнес-правило, проверяет валидность данных и передает управление в NotesRepository.
    4. Repository (Data Layer) выполняет транзакцию в Room DAO, которая обновляет локальную базу данных.
    5. Изменение в БД автоматически прокидывается обратно в UI через Flow, обновляя состояние экрана.
